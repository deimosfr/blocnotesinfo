1 Introduction

L'échange de clefs SSH est génial pour ne pas avoir à taper son mot de passe SSH. C'est d'autant plus très simple à réaliser.

2 Basique

2.1 Serveur

Sur le serveur, le compte de l'utilisateur sur lequel le client va se connecter (root par exemple), il faut que le dossier .ssh soit présent : 

mkdir .ssh

2.2 Client

Sur le client, il faut générer une paire de clef, sauf si vous en posédez déjà une (~./ssh/id_dsa.pub) :

ssh-keygen 
ssh-keygen -t rsa

Notes
Pour plus de sécurité, chiffrez votre clé privée à la création en ajoutant -p. Si vous vous faites voler vos clés, ce ne sera pas si grave :
ssh-keygen 
ssh-keygen -t rsa -p


Ensuite, il faut envoyer la clef au serveur : 

cat .ssh/id_rsa.pub | ssh root@hote_distant "cat >> .ssh/authorized_keys"

ou 

ssh-copy-id -i ~/.ssh/id_rsa.pub root@serveur-distant

Maintenant, si on se connecte sur le serveur, il ne nous demande plus de mot de passe :

ssh 
ssh root@hote_distant

2.3 Change ssh key passphrase

You can change your ssh passphrase :

ssh-keygen 
ssh-keygen -f ~/.ssh/id_rsa -p

3 Complexe avec restrictions

Si par exemple vous ne souhaitez pas que depuis n'importe où on puisse se connecter avec l'utilisateur root, il faut faire un échange de clef basique entre la machine cliente et le serveur, puis éditer sur le serveur :

  • Le fichier de configuration du server OpenSSH /etc/ssh/sshd_config :
/etc/ssh/sshd_config 
PermitRootLogin without-password

  • La clef autorisée /root/.ssh/autorized_keys :
from="10.0.0.1" ssh-dss..... (la clef en question)

Et pour finir redémarrez OpenSSH :-). Maintenant, seule la machine 10.0.0.1 sera autorisée à se connecter directement en root et uniquement via la clef.

Si vous avez plusieurs machines ou hôtes à ajoutez, séparez les par des virgules.

4 ssh-add

ssh-add[1] est un outil permettant d'avoir une clef ssh privée avec passphrase et de ne pas avoir à la taper à chaque fois, mais simplement une fois lors de la première utilisation. Il existe aussi son penchant X qui s'appelle ssh-askpass. Il est possible également de définir un timeout :

ssh-add 
ssh-agent
ssh-add -t 3600

Donc là au bout d'une heure, il faudra de nouveau rentrer la passphrase.

5 FAQ

5.1 Authentication refused: bad ownership or modes for directory

Si vous rencontrez ce type d'erreur : 

Authentication refused: bad ownership or modes for directory /home/client
Failed publickey for client from x.x.x.x port 57113 ssh2
Connection closed by x.x.x.x

Vous avez des problèmes de droit dans le home de votre utilisateur. Vérifiez bien qu'il a quelquechose comme 755. Si ce n'est pas possible de changer les droits, alors il faut dire à SSH d'être moins restrictif sur les permissions. Il faut donc modifier le fichier /etc/ssh/sshd_config et mettre cette option : 

StrictModes no

5.2 Disabling protocol version 1. Could not load host key

J'ai eu ce petit soucis notament avec le live CD de Xen Enterprise pour effectuer des P2V. Je voulais me connecter à distance pour vérifier l'état d'avancement de la migration. Or il fallait générer des clefs SSH pour démarrer le serveur. Voici donc la procédure : 

ssh-keygen -t rsa1 -f /etc/ssh_host_rsa_key -N ""
ssh-keygen -t dsa -f /etc/ssh_host_dsa_key -N ""
/usr/sbin/sshd

Et voilà le problème du serveur avec clefs manquantes est résolu :-)

5.3 Je ne peux pas changer le mot de passe root et je veux absolument me connecter sur la machine

Attention à cette tchnique car n'importe qui pourra s'y connecter. Mais pour les plus aventuriers d'entre vous, modifiez ces paramètres :

/etc/ssh/sshd_config 
PermitRootLogin yes
StrictModes no
PermitEmptyPasswords yes

Redémarrez votre service SSH, et voilà, votre serveur est maintenant une vrai passoire :-)
